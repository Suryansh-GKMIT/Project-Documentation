Functional Specification Document

Project: SkillCheck – Enclosed Access Online Test Platform (MVP)

1. Introduction

SkillCheck is an enclosed-access online testing system for internal assessments. Admins manage users, Test Creators design tests and assign them to specific users, and Test Takers log in to complete tests within a predefined duration. The MVP includes roles, test creation, question management, user assignment, timed test attempts, and result viewing. Proctoring and monitoring features are excluded from this version.

2. System Overview

Frontend: React
Backend: Node.js + Express
Database: PostgreSQL
Authentication: JWT-based authentication for Admin and Test Creators; password-based login for Test Takers
Deployment: Internal network or private cloud environment
Access Model: No public sign-up; all users managed by Admin

3. Roles and Permissions

3.1 Admin

  • Create users
  • Edit or deactivate users
  • Grant or revoke Test Creator permissions
  • View all tests and results

3.2 Test Creator

  • Create, edit, and delete tests
  • Add and manage test questions
  • Assign users to take tests
  • Publish tests to Active state
  • View results of their tests

3.3 Test Taker

  • Log in securely
  • View only assigned tests
  • Attempt tests once (unless reset)
  • Submit responses before time expires

4. Functional Requirements

4.1 Authentication and Authorization

  1. Admin and Test Creators log in using email and password.
  2. Test Takers also log in to access assigned tests.
  3. System issues JWT to authenticated Admins and Creators.
  4. Role-based access ensures only authorized users can create/edit tests.

Acceptance Criteria: Unauthorized users cannot access test creation environment.

4.2 User Management (Admin)

  1. Add new users with role = Test Creator or Test Taker.
  2. Modify existing user details.
  3. Activate or deactivate users.
  4. Grant or revoke Test Creator privileges.

Acceptance Criteria: Users update immediately; inactive users cannot log in.

4.3 Test Management (Creator)

  1. Create new tests with title, description, and duration.
  2. Save tests as Draft.
  3. Publish tests when ready.
  4. Only tests in Draft can be edited freely.
  5. Tests without attempts may be deleted.

Acceptance Criteria: A test cannot be published without at least one question.

4.4 Question Management

  1. Only MCQ questions supported in MVP.
  2. Each question must have text, 2–4 options, and a correct answer.
  3. Questions can be updated or removed while the test is in Draft.

Acceptance Criteria: All questions linked to correct test ID and stored with valid structure.

4.5 Assigning Users to Tests

  1. Creator selects users from the registered user list.
  2. Users assigned to the test are stored in a linking table.
  3. Only assigned users can attempt the test.

Acceptance Criteria: Assigned test becomes visible in Test Taker dashboard immediately after activation.

4.6 Test Attempt Behavior (Taker)

  1. Test Taker logs in and sees a list of assigned tests.
  2. Timer starts when user clicks “Start Test.”
  3. System auto-saves responses as user navigates.
  4. Test auto-submits when timer expires.
  5. One attempt allowed unless reset manually.

Acceptance Criteria: Timer must continue accurately on browser refresh.

4.7 Scoring and Results

  1. System scores MCQs based on correct answers.
  2. Creator and Admin can view results.
  3. Results must show Taker details, timestamps, responses, and score.
  4. Creator/Admin can export data in CSV.

Acceptance Criteria: Scoring algorithm is accurate and consistent.

5. Non-Functional Requirements

  1. Security: Password hashing, JWT, HTTPS, and access control.
  2. Performance: Support for multiple simultaneous Test Takers.
  3. Reliability: Auto-save to prevent data loss.
  4. Usability: Simple dashboards for each role.
  5. Data integrity: Responses and scores stored consistently.
  6. Extensibility: Future proctoring features must integrate without major redesign.

6. Data Model Overview

Users

  • id
  • name
  • email
  • password_hash
  • role (admin, creator, taker)
  • is_active

Tests

  • id
  • title
  • description
  • duration_minutes
  • status (draft, active, closed)
  • created_by

Questions

  • id
  • test_id
  • question_text
  • options (JSON array)
  • correct_option_id

Test Assignments

  • id
  • test_id
  • user_id

Attempts

  • id
  • test_id
  • user_id
  • started_at
  • ended_at
  • responses (JSON)
  • score

7. Acceptance Criteria Summary

  1. Admin can create and manage users.
  2. Test Creator can create tests and assign users.
  3. Test Taker sees only assigned tests.
  4. Timer works reliably.
  5. Auto-submit works correctly.
  6. Scoring is accurate.
  7. Results are exportable.

8. Future Scope

  1. Window/tab-switch monitoring
  2. Camera and microphone enforcement
  3. AI-based cheating detection
  4. Proctoring dashboard
  5. Mobile application
  6. Randomized question sets per user
